Cross-device Regression

跨裝置回歸測試架構

問題

單一產品需同時支援 5 個平台，雙週一次發版，人工回歸速度跟不上迭代節奏，且不同裝置測試邏輯容易分散、難維護。

做法

  • 建立跨平台共用測試邏輯層，裝置差異抽出成獨立規則檔
  • 設計可重用測試資料集，降低多語系/多裝置維護成本
  • 建立團隊標準化 SOP，帶領 13 人共同維運

Command Architecture

Purchase-Content-Flow 分層設計

問題

不同裝置購買方式不同，測試帳號與信用卡需要頻繁替換，若寫死在流程中，會造成維護困難與跨人協作誤判。

做法

  • 將 command、account profile、payment profile、device rule 拆成獨立分層
  • 採用 machine-friendly 參數命名，降低輸入格式不一致造成的錯誤
  • 設計可擴充規則，方便未來新增 refund-flow、plan-change-flow 等場景

API & Exploratory Testing

服務層 API 測試與探索性測試流程

問題

金流、訂單、帳務等服務程式若僅靠功能 UI 測試，難以覆蓋邊界條件與異常回覆；且未發布功能缺乏明確驗收標準，容易在探索性測試階段漏掉潛在影響範圍。

做法

  • 以 Postman + Python(requests) 建立 API 測試案例，涵蓋正向流程、錯誤處理、邊界條件
  • 用 SQL 查詢驗證後端資料一致性，確保 API 回覆與資料庫狀態同步
  • 對未發布功能，先定義需求與預期行為，再設計探索性測試涵蓋潛在影響範圍
  • 撰寫結構化測試計劃與測試報告，讓 PM / RD 快速掌握品質風險

Agentic AI Architecture

五層式 Agentic AI 測試架構重構

問題

單次 prompt 驅動的 AI 測試流程難以重複、難以維護，跨裝置、跨國別、跨語系時容易出現流程跑偏與誤操作，管理層也難以追蹤執行證據。

做法

  • 重新設計五層架構：核心規則層（CLAUDE.md）、命令入口層（.claude/commands/）、流程編排層（flows/）、配置資料層（devices/accounts/lang/elements）、驗證輸出層（testcases/reports/screenshots）
  • 將流程、裝置、帳號、語系、驗證標準拆開管理，降低人為輸入差異與平台耦合
  • 切換國別、帳號、裝置或語系時，只需替換設定檔，不需改動 flow 本身；新增流程（退款、訂閱、家長控制等）也只需擴充 flow/testcase，不需重寫整體架構

技術棧

Claude Code, CLAUDE.md 規則設計, Flow/Command/Testcase 分層架構, 多裝置/多國別/多語系配置管理
